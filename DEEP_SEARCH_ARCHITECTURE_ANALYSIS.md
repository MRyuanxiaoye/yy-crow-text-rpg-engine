# 深度搜索智能体（Deep Search Agent）架构分析报告

## 一、 核心目标：构建“资深中国文化研究者”

我们的目标不仅仅是一个搜索引擎，而是一个具备以下特征的智能体：
1.  **资深认知**：拥有深厚的领域知识，能识别专业术语和隐性关联。
2.  **核心洞察**：能从宽泛的问题中精准定位核心议题（The "Heart" of the question）。
3.  **深度挖掘**：像老师一样，不仅给出结论，还深入剖析原理、历史演变和关键细节。
4.  **自动发散**：在讲透核心的基础上，适度延伸相关知识点，构建知识网络。

## 二、 痛点与底层逻辑拆解

### 1. 资料数据支撑
*   **痛点**：本地存储有限，爬虫抓取可能因关键词不准而失效。
*   **解决方案**：**“元数据索引” + “推理式搜索”**
    *   **元数据索引**：本地只存目录、书目、核心概念表（Entity Graph），不存全文。
    *   **推理式搜索**：LLM 不直接搜索问题，而是先基于元数据生成“搜索策略”。例如，不知道某符箓画法，但知道它在《道藏》某部，则生成指令“搜索《正统道藏》洞神部关于XX的内容”。

### 2. 关键词提取与问题解析
*   **痛点**：关键词提取不准，导致方向错误。
*   **解决方案**：**Query Expansion（查询扩展） & Rewriting（重写）**
    *   **多维扩展**：利用 LLM 的领域知识，将用户口语转化为专业术语（如“内阁制度” -> “票拟”、“批红”、“文渊阁”）。
    *   **意图消歧**：区分用户是想看“故事”还是“学术考证”。

### 3. 核心定位（找准方向）
*   **痛点**：找不准核心，后续步骤全部跑偏。
*   **解决方案**：**Planner-Executor（规划-执行）架构**
    *   **慢思考（Slow Thinking）**：在执行搜索前，先强制进入规划阶段。
    *   **生成大纲**：LLM 先输出一份研究大纲（Outline），明确要研究的 3-5 个核心子维度。
    *   **人机对齐**：可选步骤，将大纲展示给用户确认，确保方向一致。

### 4. 深度挖掘（像老师一样讲解）
*   **痛点**：回答浅显，缺乏细节，一笔带过。
*   **解决方案**：**Reflexion（反思机制） & Iterative Refinement（循环优化）**
    *   **Critique（批判）**：引入 Reviewer 角色，检查初步回答是否包含具体年代、人物、原文引用。
    *   **Loop（循环）**：如果 Critic 认为不够深，触发二次搜索（Deep Digging），专门查找缺失的细节。
    *   **Inference（推理补充）**：允许 LLM 基于训练数据对缺失信息进行逻辑补全，并在报告中注明。

### 5. 延伸发散（知识网络）
*   **痛点**：发散容易跑题，深度难以把控。
*   **解决方案**：**受控发散（Bounded Exploration）**
    *   **图谱思维**：基于实体的 1 跳（1-hop）邻居进行发散。
    *   **分离展示**：在正文讲透核心后，单独设立“知识延伸”板块，简要介绍相关概念，不干扰主线。

## 三、 技术架构升级路线 (Python Script -> AI Agent)

从当前的线性脚本升级为 **Multi-Agent System (多智能体系统)**。

### 核心模块重构
1.  **Planner Agent**：负责拆解问题，生成动态搜索大纲。
2.  **Researcher Agent**：负责执行具体的搜索任务，具备“自我纠错”能力（搜不到换个词搜）。
3.  **Reviewer Agent**：负责阅读 Researcher 的成果，判断是否满足“深度”要求，提出改进意见。
4.  **Writer Agent**：负责最终的报告撰写，确保风格如“资深研究员”。

### 推荐技术栈
*   **框架**：LangGraph (Python) - 专为构建有状态、循环的 Agent 设计。
*   **LLM**：DeepSeek-V3 / GPT-4o (负责逻辑规划), Llama 3 (负责本地处理)。
*   **Search**：DuckDuckGo / Google Serper API。
*   **Database**：SQLite (元数据), ChromaDB (临时向量内容)。
