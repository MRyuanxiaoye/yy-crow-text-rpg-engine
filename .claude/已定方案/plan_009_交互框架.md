## 零、序幕→首次交互的无缝衔接

### 核心原则
序幕结束后不发数据面板，直接触发第一个NPC来访（或序幕本身包含首个NPC的发言）。

### 节奏要求
- 序幕（场景+在场NPC描写）→ 首个NPC开口说事 → 玩家回应，全程无断裂
- 数据面板改为玩家主动请求时才展示（如输入'查看状态'或按钮），不作为默认流程
- 如果序幕场景中已有NPC在场，序幕文本应以NPC发言结尾，直接进入对话

### 交互示例
明末：
> 崇祯元年八月，乾清宫东暖阁。先帝批红的朱砂笔还搁在笔架上，墨迹已干透。
> 温体仁垂手站在左侧，王承恩躬身在右手捧奏疏，孙承宗白发如雪站在御案正前方。
> 温体仁先开口：陛下，厂卫报说，魏忠贤的干儿子们昨夜在天津卫烧了半屋子文书。
> 孙承宗抬眼：陛下，老臣在辽东时学会一件事——火起了，先看风向。
> *💡 可选方向：追问文书内容 / 问孙承宗何意 / 让王承恩念奏疏*
> 三双眼睛，都在等您开口。

盗墓：
> 长白山深处，你蹲在半塌的墓道口，手电光在潮湿的石壁上晃。
> 胖子在后面嘟囔：三儿，这味儿不对，像是新翻过的土。
> 闷油瓶已经无声滑进了墓道，只留下一句：有人来过。最近三天。
> *💡 可选方向：跟进去 / 先检查墓道口痕迹 / 问胖子土味判断*

## 一、NPC主动来访系统

### 核心范式转变
从'玩家看仪表盘主动召见NPC'变为'NPC带着事情主动来找玩家'。

### 五种触发源
1. 议程驱动：回合议程中的事项，由相关NPC携带来访
2. 延迟结算：上回合决策的后果反馈
3. 维度阈值：某维度到达危险值，相关NPC紧急来报
4. 被动事件：剧本预设的随机/条件事件
5. NPC自身动机：NPC基于性格底色主动求见

### 来访队列
- 按紧急度排序（维度严重程度 + 事件危急程度）
- 玩家可以拒绝接见（后果自负）、打断当前来访、插队召见
- 来访形式根据NPC距离：亲自到访 / 传令兵 / 书信 / 间接传闻

### NPC位置约束
- 来访形式必须尊重NPC的地理位置
- NPC的location字段决定其能否亲自到场：
  - 与玩家同地点 → in_person（亲自到访）
  - 同区域不同地点 → in_person（需等候片刻）
  - 远方（如辽东、边关） → messenger（信使/传令兵）或 letter（书信/塘报）
- messenger/letter形式时，到场的是信使而非NPC本人
- 序幕场景同样遵守此约束：不在当前地点的NPC不出现在物理场景中

### 交互示例

明末：
> 太监来报：陛下，兵部尚书孙传庭求见，说辽东军情紧急。
> *💡 辽东欠饷三月，此事可能涉及调兵和军饷两个议题*
> 玩家输入：宣。

远程NPC示例（明末）：
> 传令兵跪地呈上塘报：陛下，辽东督师袁崇焕八百里加急！
> 塘报内容：宁远粮道已断三日，鞑子铁骑压境，请拨军饷四十万两，否则恐城垣不保。
> *💡 此事涉及军饷调度，可召户部尚书毕自严参议*

盗墓：
> 胖子从隔壁墓室跑过来：三儿！你快来看，这墙上有字！
> *💡 壁画上似乎有西域文字，可能与墓主身份有关*
> 玩家输入：走，去看看。

## 二、GM交互层

### 三个功能
1. **框定（Frame）**：在决策点给出可选方向提示
2. **注释（Annotate）**：基于玩家维度的感知补充（维度越高看到越多细节）
3. **警示（Alert）**：高风险操作前的警告
4. **记录（Record）**：实时捕获对话中的玩家决策并反馈确认

### 核心原则
- 所有GM内容都是**提示文字**，不是按钮
- 玩家始终可以自由输入任何内容，不受提示限制
- 维度门控是连续渐变，不是二元开关

### 交互示例

明末：
> 孙传庭：陛下，辽东欠饷三月，恐生兵变。
> *💡 可选方向：追问具体数额 / 询问能否以粮代饷 / 问责户部*
> *👁 [军事 > 60] 你注意到孙传庭提到'恐生兵变'时语气刻意加重——辽东的情况可能比他说的更严重*
> 玩家自由输入...

盗墓：
> 你发现石门上有两个锁孔和一行小字。
> *💡 可以试着开锁 / 仔细辨认小字 / 叫老陈来看看*
> *⚠ 这类双锁机关在西周墓中常有连锁陷阱，贸然开启风险极高*

### 4. 决策实时捕获与反馈闭环

#### 问题
当前架构中，玩家决策只在对话结束时由_summarize_talk提取。如果对话持续多轮不结束，所有决策都不会进入系统——backlog为空、agenda不更新、NPC不知道已决事项。

#### 方案：每轮GM扫描 + 决策卡片反馈

每次NPC回复后，执行一次轻量GM扫描（~100-200 token输入），判断玩家是否在本轮做了决策。

流程：
```
玩家发言 → NPC回复 → GM轻量扫描 → 检测到决策？
                                      ├─ 否 → 继续对话
                                      └─ 是 → 
                                            ├─ 决策卡片插入对话流（玩家看到确认）
                                            ├─ 更新world_memory
                                            ├─ 标记agenda为addressed
                                            ├─ 如需执行 → 进backlog（instant/delayed）
                                            └─ 注入后续NPC上下文（已决事项列表）
```

#### 决策卡片
当GM扫描检测到决策时，在NPC回复之后插入一张决策卡片：
- 决策内容（一句话描述）
- 预期影响（维度变化预估，精度受智谋门控）
- 执行方式（即时 / delayed N回合）
- 关联议程（哪个agenda被处理）
- 两个按钮：[撤回决定] [确认并继续]

玩家点[确认]后决策写入state；点[撤回]则不写入，当作没说过。

#### 已决事项注入NPC上下文
决策确认后，后续NPC prompt中注入【已决事项】段落：
```
【已决事项】
- 魏党处置：诛首恶、胁从退赃（第2回合决定）
- 辽东军饷：以魏党所得优先拨付（第10回合决定）
```
NPC看到已决事项后自然不再重复该话题。

#### 决策记录可查
累积的决策记录在行动面板/棋盘的'进行中'区域可见，也可通过/决策记录命令查看。

#### GM扫描prompt设计原则
- 只送最近1轮对话（玩家发言+NPC回复）+ 当前agenda列表 + 已决事项列表
- 输出结构：{has_decision: bool, decision_summary: str, related_agenda_id: str, execution_type: instant/delayed, predicted_effects: {...}}
- 如果has_decision为false，不做任何后续操作

#### 支线任务质量保障
GM扫描同时负责支线任务的注册质量：
- 注册前校验name和description不为空
- 对比已有支线避免重复注册（相似度检查）
- 玩家已处理的支线标记为completed

## 三、对话中途召见NPC

### 核心体验
玩家在与NPC对话过程中，发现需要另一个NPC的专业意见或权限，可以直接召来加入当前对话，无需中断重来。

### 1. 触发方式：自然语言 + GM提示
玩家随时可自由输入：'把户部尚书叫来' / '让老陈过来' / '我需要懂机关的人'

GM在合适时机给出提示（不是按钮）：
- 明末：*💡 此事涉及户部调度，可召户部尚书毕自严入殿参议*
- 盗墓：*💡 闷油瓶或许能辨认这些文字，可以喊他过来*

### 2. 加入方式：根据NPC距离决定
| 场景 | 加入形式 | 时间成本 |
|------|----------|----------|
| 同一地点 | 直接走过来，即时加入 | 无 |
| 同一区域 | 需要等一小会，GM描述过渡 | 可能推进少量时间 |
| 远距离 | 无法当场召来，只能传信 | GM提示此人不在附近 |

### 3. 多人对话结构
- 新NPC加入时获得上下文简报（由在场NPC简述或GM生成），不假装听到全部对话
- 被召NPC优先发言
- 玩家可指定对谁说话，未指定时按话题相关性决定谁回应
- 每个NPC独立上下文：全程参与者=全部历史，中途加入者=简报+加入后内容

### 4. 离场机制
- 玩家明确让其退下
- 话题转移GM提示离场
- 回合结束自动清场（主动来访NPC除外）

### 5. 通用框架抽象
当前对话 = { 发起者, 在场NPC列表(动态增减), 每个NPC的上下文范围 }
召见触发：距离检查 → 入场叙事 → 上下文注入 → 发言权重调整

## 全局交互原则
1. GM所有选项/建议都是提示文字，不是按钮，玩家始终自由输入
2. 所有交互行为都有叙事解释，不凭空出现消失
3. 通用引擎设计，不绑定特定剧本
4. 对话中的决策必须实时捕获并反馈确认，形成'决定→确认→影响→推进'闭环
