## 一、整体架构概览

深度搜索采用 **LangGraph 状态图** 驱动的流水线架构，核心流程如下：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ check_cache │───▶│   planner   │───▶│ researcher  │───▶│   reader    │───▶│   writer    │
│  缓存检查    │    │  智能规划    │    │  并发搜索    │    │  深度阅读    │    │  报告生成    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
       │                                                                            │
       │ (命中缓存)                                                                  │
       └────────────────────────────────────────────────────────────────────────────┘
                                      直接返回
```

## 二、各节点详细分析

### 2.1 缓存检查节点 (check_cache)

**位置**: `graph.py:52-72`

**功能**:
- 检查 Supabase 云端知识库是否有缓存
- 命中则直接返回，跳过后续流程

**当前实现**:
```python
cached_report = kb_manager.check_cache(query)
if cached_report:
    return {"final_report": cached_report, ...}
```

**问题**: 缓存匹配策略过于简单，仅做字符串精确匹配

### 2.2 规划节点 (planner)

**位置**: `agents/planner.py:36-320`

**核心流程** (V5 版本):

```
Step 1: 实体分析 (Entity Analysis)
    ├── 识别核心实体 (focus_entity)
    ├── 判断实体类型 (person/event/concept/thing/organization)
    ├── 提取相关实体 (related_entities, 最多3个)
    └── 判断领域 (domain)

Step 2: 框架融合 (Framework Fusion)
    ├── 根据实体类型选择主框架 (primary_framework)
    ├── 根据相关实体类型选择次框架 (secondary_framework)
    └── 融合维度：主框架全部维度 + 次框架各取2个维度

Step 3: 生成主查询 (Primary Queries)
    ├── 第一个查询必须是 "{focus_entity} 全貌背景概述"
    └── 每个主维度生成 1 个宏观搜索词

Step 4: 深度挖掘 (Deep-Dive)
    ├── 对每个主维度（跳过第一个背景概述）
    └── 生成 2 个更深入的子搜索词

Step 5: 生成次查询 (Secondary Queries)
    ├── 每个次维度生成 1 个浅层搜索词
    └── 标记 priority="secondary"
```

**分析框架** (`planning_frameworks.py`):

| 实体类型 | 框架名称 | 维度数量 |
|---------|---------|---------|
| person (历史) | 历史人物深度分析框架 | 5 |
| person (通用) | 人物分析框架 | 4 |
| event (历史) | 历史事件深度分析框架 | 5 |
| concept | 概念解析框架 | 4 |
| why | 深度因果分析框架 | 4 |
| how | 过程与方法分析框架 | 4 |

**输出示例**:
```python
{
    "domain": "历史人物深度分析框架（融合）",
    "focus_entity": "安禄山",
    "focus_type": "person",
    "related_entities": ["安史之乱", "唐玄宗", "杨贵妃"],
    "sub_topics": [
        {"sub_query": "安禄山 全貌背景概述", "priority": "primary"},
        {"sub_query": "安禄山 性格特点与内在动机", "priority": "primary"},
        {"sub_query": "安禄山早年如何获得唐玄宗信任", "priority": "primary", "parent_dimension": "..."},
        ...
        {"sub_query": "安史之乱的背景与起因", "priority": "secondary"},
    ]
}
```

### 2.3 搜索节点 (researcher)

**位置**: `agents/researcher.py:21-307`

**搜索引擎矩阵**:

| 引擎 | 定位 | 特点 | 并发限制 |
|-----|------|------|---------|
| Exa | 神经搜索 | 英文深度、语义理解强 | 3 并发 |
| Bocha | 生态搜索 | 中文精准、国内资源 | 2 并发 |
| Tavily | 事实搜索 | 全球覆盖、速度快 | 无限制 |
| DuckDuckGo | 备用 | 兜底方案 | - |

**分层搜索策略**:

```
Primary 查询 (priority="primary"):
    └── 全引擎搜索: Exa + Bocha + Tavily

Secondary 查询 (priority="secondary"):
    └── 轻量搜索: 仅 Tavily
```

**搜索结果处理**:
1. 多引擎结果合并
2. LLM 提取核心事实
3. 标记长文 URL (`FOUND_LONG_TEXT: <URL>`)

### 2.4 阅读节点 (reader)

**位置**: `tools/reader_engine.py:17-280`

**四层混合阅读器**:

```
Level 1: 爬虫抓取
    ├── aiohttp + trafilatura 提取正文
    └── Tavily 兜底（反爬场景）

Level 2: BM25 关键词初筛
    ├── jieba 分词
    └── 从 N 个片段选 20 个

Level 3: 向量语义精筛
    ├── OpenAI text-embedding-3-small
    ├── ChromaDB 临时向量库
    └── 从 20 个片段选 5 个

Level 4: LLM 深度总结
    └── 基于 5 个精选片段生成摘要
```

### 2.5 写作节点 (writer)

**位置**: `agents/writer.py:5-100`

**核心理念**: 瀑布式知识倾泻

**写作原则**:
1. **一泻而下**: 不做中途小结，持续深入
2. **资料决定长度**: 不设上限/下限
3. **因果链条不断**: 每个事实追问"为什么"
4. **精彩故事突出概括**: 关键节点串联，不压缩成无趣的话

**结构设计**:
```
第一层: 直接回答（时间、定义、核心答案）
第二层: 背景展开（为什么会有这个问题）
第三层: 机制拆解（具体怎么运作）
第四层: 关键节点（决定性人物/事件）
第五层+: 深入再深入...
```

**禁止事项**:
- ❌ 文末总结
- ❌ 过渡词（"值得注意的是"、"更重要的是"）
- ❌ 固定段落数
- ❌ 感叹号和营销号句式

## 三、数据流状态定义

```python
class AgentState(TypedDict):
    # 输入
    query: str

    # Planner 输出
    domain: Optional[str]
    role_name: Optional[str]
    persona_instruction: Optional[str]
    search_strategy: Optional[str]
    output_format_instruction: Optional[str]
    sub_topics: List[dict]

    # 数据累积
    research_data: Annotated[List[str], operator.add]
    long_text_urls: Annotated[List[str], operator.add]

    # 最终输出
    final_report: str
    keywords: List[str]
    logs: Annotated[List[str], operator.add]

    # 成本追踪
    total_tokens: Optional[int]
    total_cost: Optional[float]
```

## 四、当前架构的优势

1. **实体感知规划**: 能识别核心实体类型，选择对应分析框架
2. **分层搜索策略**: 主查询深度搜索，次查询浅层覆盖，避免信息过载
3. **混合阅读器**: BM25 + 向量双重筛选，平衡效率与质量
4. **瀑布式写作**: 层层深入，因果链条连贯
5. **异步并发**: 搜索和阅读均支持异步，提升效率

## 五、当前架构的不足

### 5.1 规划层面

| 问题 | 描述 |
|-----|------|
| **缺乏问题核心分析** | 直接跳到实体识别，未先分析"用户真正想知道什么" |
| **维度含义不清晰** | 框架维度是预设的，未针对具体问题解释每个维度的意义 |
| **维度间关系缺失** | 各维度独立生成，未建立维度间的逻辑关联 |
| **扩散能力弱** | 仅通过 related_entities 做有限扩散，缺乏主动发散 |

### 5.2 搜索层面

| 问题 | 描述 |
|-----|------|
| **搜索词质量依赖 LLM** | 搜索词由 LLM 生成，质量不稳定 |
| **缺乏搜索结果评估** | 未评估搜索结果是否真正回答了维度问题 |

### 5.3 写作层面

| 问题 | 描述 |
|-----|------|
| **维度整合不足** | Writer 收到的是扁平化的 research_data，丢失了维度结构 |
| **缺乏核心-维度-答案的映射** | 最终报告未明确展示"核心问题→维度→发现"的逻辑链 |
