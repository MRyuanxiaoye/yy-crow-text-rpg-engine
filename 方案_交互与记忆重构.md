# 方案：交互模式与记忆系统重构

## 设计目标

1. 以玩家与NPC直接对话为主要交互方式，旁白退居辅助角色
2. 通过结构化长期记忆保证剧情一致性，不依赖纯上下文窗口
3. 用飞书卡片按钮驱动流程，降低玩家操作门槛

---

## 一、交互模式重构

### 1.1 两种状态

| 状态 | 说明 | 玩家可做的事 |
|------|------|-------------|
| 行动面板（idle） | 不在对话中，旁白展示局势 + 按钮 | 只能点按钮选择动作 |
| 对话模式（talking） | 正在和某个NPC交谈 | 自由打字，无回合限制 |

### 1.2 行动面板

每次对话结束 / 结算完成后，旁白发送一张卡片：
- 当前时间 + 核心数值
- 近况简述（1-2句）
- 按钮列表：当前可召见的NPC + 「查看局势」

按钮点击 → 服务端收到回调 → 切换到对话模式（不需要玩家手动输入指令）。

### 1.3 对话模式流程

```
玩家点击「对话温体仁」
  → 系统切换 state.phase = "talking", state.talking_to = "温体仁"
  → 后续玩家消息全部路由给温体仁回复
  → NPC回复卡片底部始终带「结束对话」按钮

玩家自由对话（无回合限制）
  → NPC根据人格卡 + 记忆 + 世界状态回复
  → 如果NPC提出可执行建议 → 卡片附带多选决策选项

玩家点击「结束对话」
  → 触发记忆提取（本次对话关键事实写入NPC记忆）
  → 检查是否有已勾选的决策
    → 有决策 → 旁白结算（叙事 + 数值 + 时间推进 + 事件）
    → 无决策 → 旁白简短总结 + 行动面板

玩家点击「确认决策」（对话中勾选了选项）
  → 等同于「结束对话 + 执行决策」，自动退出对话模式
  → 旁白结算
```

### 1.4 决策多选卡片

NPC回复中包含可执行建议时，卡片底部附带：

```
┌─────────── 决策选项 ───────────┐
│ ☐ 选项A                       │
│ ☐ 选项B                       │
│ ☐ 选项C                       │
│                                │
│ [全选]        [确认决策]        │
└────────────────────────────────┘
```

- 玩家可以不选，继续聊
- 勾选后点「确认决策」→ 自动结束对话 + 进入结算
- 决策选项由AI从NPC回复中实时提取

### 1.5 旁白的职责（仅以下场景出场）

- 对话结束后的结算叙事（行动反馈 + 数值变化 + 时间推进）
- 事件播报（天灾、NPC主动行为等外部事件）
- 章节过渡
- 行动面板展示

### 1.6 切换NPC

必须先结束当前对话，再选新NPC。避免上下文混乱，保证每段对话有明确边界。

## 二、记忆系统

### 2.1 架构总览

```
短期记忆：最近对话原文（当前对话的完整内容）
长期记忆：
  ├── NPC独立记忆（每个NPC各自的事实列表）
  ├── 全局世界状态（world_summary，200字文字描述）
  ├── 章节摘要（每章结束时压缩）
  └── 承诺追踪（玩家未兑现的承诺）
```

### 2.2 NPC独立记忆

**数据结构**（存储在 state.npc_memories 中）：

```yaml
npc_memories:
  温体仁:
    - fact: "玩家承诺让他代理阁务"
      turn: 12
      chapter: "ch01"
      sentiment: "grateful"
    - fact: "玩家当面斥责他结党"
      turn: 25
      chapter: "ch02"
      sentiment: "fearful"
  王承恩:
    - fact: "玩家要求他彻查宫中内奸"
      turn: 8
      chapter: "ch01"
      sentiment: "loyal"
```

**触发时机**：每次与该NPC对话结束时（点击「结束对话」或「确认决策」）。

**提取方式**：调用Claude，输入本次对话全文，输出结构化事实列表。

**注入方式**：下次与该NPC对话时，将其全部memories注入NPC的system prompt。

**压缩机制**：单个NPC记忆超过15条时，调用Claude将早期记忆合并压缩（保留关键事实，合并细节）。

### 2.3 全局世界状态（world_summary）

**数据结构**：一段200字以内的文字描述，存储在 state.world_summary 中。

```
当前局势：温体仁得势，周延儒被罢。朝中无人敢言，但周延儒门生暗中串联。
辽东军饷已拖欠两月，吴三桂态度暧昧。李自成在陕西势力日壮。
玩家未兑现的承诺：曾答应王承恩彻查宫中内奸，至今未动。
```

**触发时机**：每次旁白结算后（时间推进 + 数值变化时），调用Claude重新生成。

**输入**：当前数值、最近决策、NPC态度变化、活跃事件。

**注入方式**：所有NPC对话和旁白叙事的system prompt都带上这段。

### 2.4 章节摘要（chapter_summaries）

**数据结构**：

```yaml
chapter_summaries:
  ch01:
    summary: "玩家罢免周延儒，扶温体仁上位。拨款赈灾但国库吃紧。拒绝了吴三桂增兵请求。"
    key_decisions: ["罢免周延儒", "赈灾拨款", "拒绝增兵"]
    npc_attitude_shifts:
      温体仁: "感恩戴德 → 权欲膨胀"
      吴三桂: "恭顺 → 不满"
```

**触发时机**：章节切换时，调用Claude对整章历史做压缩总结。

**注入方式**：后续章节的system prompt带上所有前置章节的摘要。

### 2.5 承诺追踪（promises）

**数据结构**：

```yaml
promises:
  - id: "p001"
    content: "答应王承恩彻查宫中内奸"
    made_to: "王承恩"
    turn: 8
    chapter: "ch01"
    status: "pending"    # pending / fulfilled / broken
  - id: "p002"
    content: "答应温体仁代理阁务"
    made_to: "温体仁"
    turn: 12
    chapter: "ch01"
    status: "fulfilled"
```

**触发时机**：
- 提取：对话结束时的事实提取同时识别承诺
- 更新：结算时检查是否有承诺被兑现（Claude判断）
- 超时：承诺pending超过一定回合数，NPC态度下降

**注入方式**：与对应NPC对话时，pending状态的承诺注入prompt（NPC会主动提起"陛下上次答应臣的事……"）。

## 三、Prompt注入策略

### 3.1 NPC对话时的prompt组成

```
[system]
1. NPC人格卡（固定）
2. 全局世界状态 world_summary（~200字）
3. 该NPC的memories（全部事实）
4. 与该NPC相关的pending承诺
5. 当前数值概况（一行）

[messages]
6. 本次对话的完整消息记录（短期记忆）
```

### 3.2 旁白叙事时的prompt组成

```
[system]
1. 旁白角色设定
2. 全局世界状态 world_summary
3. 当前章节摘要 + 前置章节摘要
4. 当前活跃事件
5. 数值详情

[messages]
6. 最近的对话摘要（不是原文，是提取后的事实）
```

### 3.3 Token预算估算

| 组件 | 预估token | 频率 |
|------|-----------|------|
| NPC人格卡 | 300-500 | 每次对话 |
| world_summary | 200-400 | 每次对话/叙事 |
| NPC memories（15条内） | 300-600 | 每次对话 |
| 章节摘要（累计） | 200-800 | 每次叙事 |
| 承诺列表 | 100-200 | 每次对话 |
| 本次对话原文 | 1000-3000 | 实时 |
| **总计** | **~2000-5000** | - |

在DeepSeek和Claude的上下文窗口内完全可控。

## 四、技术实现要点

### 4.1 GameState新增字段

```python
# 对话模式
talking_to: str = ""                          # 当前对话NPC，空串=不在对话中
current_talk_history: list[dict] = []         # 本次对话的完整消息

# 长期记忆
npc_memories: dict[str, list[dict]] = {}      # NPC独立记忆
world_summary: str = ""                       # 全局世界状态描述
chapter_summaries: dict[str, dict] = {}       # 章节摘要
promises: list[dict] = []                     # 承诺追踪
```

### 4.2 新增函数

| 函数 | 触发时机 | 模型 | 作用 |
|------|----------|------|------|
| extract_facts() | 对话结束 | Claude | 从对话中提取事实+承诺 |
| update_world_summary() | 结算后 | Claude | 重新生成世界状态描述 |
| generate_chapter_summary() | 章节切换 | Claude | 压缩整章为摘要 |
| compress_npc_memory() | 记忆>15条 | Claude | 合并早期记忆 |
| check_promises() | 结算时 | Claude | 检查承诺是否被兑现 |

### 4.3 飞书卡片交互

- 按钮回调：飞书卡片action → POST到服务端 → 根据payload路由
- 多选决策：checkbox组件 + submit按钮
- 需要实现卡片模板：行动面板卡片、NPC回复卡片（带结束按钮）、决策多选卡片、结算卡片

## 五、与现有方案B的关系

记忆系统是对方案B的增强，不是替代：
- 方案B解决的是"什么时候结算"→ 对话结束后结算
- 记忆系统解决的是"结算时AI知道多少"→ 注入长期记忆保证一致性
- 交互重构解决的是"玩家怎么触发"→ 按钮驱动，对话为主

三者组合后的完整流程：
1. 行动面板（按钮）→ 进入对话
2. 自由对话（NPC记忆注入）→ 沉浸交互
3. 结束对话 → 事实提取 + 记忆更新
4. 有决策 → 旁白结算（world_summary更新）
5. 回到行动面板
