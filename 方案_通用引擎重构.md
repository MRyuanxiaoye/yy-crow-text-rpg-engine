# 方案：通用文字RPG引擎重构

## 设计目标

构建一个通用的文字RPG引擎框架，实现：
- 玩家体验纯自由对话，不被系统打断
- 世界在后台自转，主动制造压力和变化
- 数值是基础骨架，事件是驱动力
- 同一套引擎可适配不同题材剧本（历史/探险/玄幻）

---

## 一、通用引擎层

### 1. 数值系统

每个剧本自定义数值维度，引擎提供通用的增减、阈值触发、自然衰减机制。

```yaml
# manifest.yaml 中定义
metrics:
  dimensions:
    - id: treasury
      name: 国库
      initial: 40
      min: 0
      max: 100
      thresholds:
        - below: 20
          trigger_event_tag: "financial_crisis"
        - above: 80
          trigger_event_tag: "prosperity"
    - id: military
      name: 军力
      initial: 50
      min: 0
      max: 100
      thresholds:
        - below: 30
          trigger_event_tag: "mutiny_risk"
```

**结算方式：** 实时结算。玩家做出决策后数值立刻变化，不攒批。

**自然衰减：** 每次时间推进时，按 `decay_rules` 自动扣减（模拟维持成本）。

---

### 2. 时间系统

```yaml
# manifest.yaml 中定义
time:
  unit: "月"
  unit_label: "崇祯{year}年{month}月"   # 显示格式模板
  start: { year: 1, month: 8 }           # 崇祯元年八月
  turns_per_unit: 3                       # 每3个玩家回合，推进1个时间单位
  decay_rules:
    - metric: treasury
      delta: -3
    - metric: border_defense
      delta: -1
```

**核心规则：**
- 每 N 个玩家回合（剧本自定义），时间推进 1 个单位
- 时间推进时触发：自然衰减 → 阈值检查 → 事件调度
- 每次旁白必须告知玩家当前游戏时间

**不同题材示例：**
| 题材 | 时间单位 | N回合/单位 | 跨度 |
|------|----------|-----------|------|
| 明末 | 月 | 3 | 16年 |
| 盗墓 | 小时 | 2 | 数天 |
| 斗破苍穹 | 年 | 4 | 数十年 |

### 3. 事件调度器

#### 3.1 事件类型

| 类型 | 说明 | 触发方式 | 示例 |
|------|------|----------|------|
| timeline | 主线锚点，必然发生 | 时间到达 | 己巳之变 |
| random | 随机变数 | 概率掷骰 | 地方官贪腐案 |
| disaster | 天灾/不可抗力 | 时间锚点或概率 | 黄河决口 |
| npc_action | NPC主动行为 | NPC状态+数值条件 | 温体仁密谋弹劾 |
| chain | 前置事件的后果 | 前置事件完成后 | 处置魏党→余党反扑 |

#### 3.2 事件通用结构

```yaml
event:
  id: "E_jisi_invasion"
  type: "timeline"
  name: "己巳之变"
  chapter: "ch02_jisi"          # 所属章节（可选，用于过滤）

  # --- 触发条件 ---
  trigger:
    time_anchor: { year: 2, month: 11 }     # timeline类型
    conditions:                              # 条件触发
      - metric: border_defense
        operator: below
        value: 30
    probability: 0.3                         # random类型，每次时间推进掷骰
    prerequisite_events: ["E_xxx"]           # chain类型
    cooldown_time_units: 3                   # 触发后冷却N个时间单位

  # --- 事件内容 ---
  payload:
    severity: "critical"                     # minor | moderate | critical
    description: "后金绕道蒙古，兵锋直逼京师"
    forced_npcs: ["皇太极"]
    narrative_hint: "八百里加急军报送入宫中"  # 给AI生成叙事的引导
    player_options_hint:                     # 暗示方向（不直接展示给玩家）
      - "固守待援"
      - "出城截击"
      - "议和"

  # --- 不处理的后果 ---
  default_consequence:
    delay_time_units: 2                      # 2个时间单位不处理则生效
    effects:
      metrics: { border_defense: -15, morale: -10 }
      narrative: "京畿沦陷，生灵涂炭"
      chain_events: ["E_jisi_aftermath"]     # 可触发连锁事件

  # --- 处理判定 ---
  resolution:
    judge_model: "claude"
    effect_range:
      metrics_delta_max: 20
      metrics_delta_min: -20
```

#### 3.3 调度规则

- **同时活跃事件上限：3个**（新事件排队）
- **优先级：** critical > moderate > minor，同级按时间先后
- **每次时间推进时执行：**
  1. 检查 timeline 事件是否到达时间锚点
  2. 检查 disaster 事件是否到达或概率命中
  3. 对 random 池掷骰
  4. 检查 npc_action 条件
  5. 检查 chain 事件前置是否满足
  6. 检查活跃事件倒计时，到期执行 default_consequence
- **冷却机制：** 同一事件触发后进入冷却，同 tag 事件共享冷却

### 4. 意图识别

引擎代码逻辑，不需要剧本配置。

**流程：**
1. 玩家发言 → DeepSeek 判断是否包含游戏行为
2. 如果是纯闲聊 → 正常对话回复，不触发机制
3. 如果包含决策/行动 → 提取意图，交给 Claude 判定数值影响
4. 无论是否有行动，回合计数 +1，累积到 N 时触发时间推进

**意图分类：**
- 决策类：下令、裁决、选择方向
- 行动类：调兵、拨款、派人
- 探索类：打探消息、调查
- 社交类：拉拢、威胁、谈判

---

### 5. 叙事生成

**旁白输出规范：**
- 每次旁白开头标注当前时间：`【崇祯二年十一月】`
- 事件触发时以叙事方式插入（NPC闯入、急报传来），不用系统弹窗
- 关键数值变化时主动展示：`（国库 40→25，军力 50→45）`
- 阈值触发时加重描写（"户部尚书面如死灰"而不只是"国库低于20"）

---

### 6. 结局判定

```yaml
endings:
  - id: "zhongxing"
    name: "中兴之主"
    conditions:
      metrics_min: { treasury: 60, military: 60, morale: 60, stability: 60 }
      required_decisions: []
    priority: 1
    narrative: "史书载：帝承大统十六年，内清阉党、外御强虏..."

  - id: "wangguo"
    name: "亡国之君"
    conditions:
      metrics_max: { stability: 20 }    # 任一核心数值崩溃
      any_below_zero: [treasury, military, morale]
    priority: 10
    narrative: "甲申三月十九，城破..."
```

**判定时机：** 最终章结束时，或任一数值触及极端阈值时提前触发。

## 二、剧本数据层

### 目录结构

```
scripts/{script_id}/
├── manifest.yaml              # 元信息：数值、时间、角色、全局设定
├── timeline.yaml              # 主线锚点事件（按时间排列）
├── events/
│   ├── random_pool.yaml       # 随机事件池
│   ├── disasters.yaml         # 天灾/不可抗力事件池
│   └── npc_events.yaml        # NPC主动行为事件池
├── npcs/
│   ├── {npc_name}.yaml        # NPC人物卡
│   └── ...
├── chapters/
│   ├── ch01_xxx.yaml          # 章节定义（大幕）
│   └── ...
└── endings/
    └── endings.yaml           # 所有结局
```

### manifest.yaml 完整示例

```yaml
script_id: mingmo
name: 明末救国
description: "崇祯年间，内忧外患，你能否力挽狂澜？"

# 数值系统
metrics:
  dimensions:
    - { id: treasury, name: 国库, initial: 40 }
    - { id: military, name: 军力, initial: 50 }
    - { id: morale, name: 民心, initial: 45 }
    - { id: stability, name: 朝廷稳定, initial: 35 }
    - { id: border, name: 边防, initial: 55 }
  thresholds:
    - { metric: treasury, below: 20, trigger_tag: "financial_crisis" }
    - { metric: military, below: 30, trigger_tag: "mutiny_risk" }
    - { metric: morale, below: 20, trigger_tag: "civil_unrest" }
    - { metric: border, below: 25, trigger_tag: "invasion_imminent" }

# 时间系统
time:
  unit: "月"
  display_format: "崇祯{year}年{month}月"
  start: { year: 1, month: 8 }
  turns_per_unit: 3
  decay_rules:
    - { metric: treasury, delta: -2 }
    - { metric: border, delta: -1 }

# 可选角色
roles:
  - { id: emperor, name: 崇祯帝, description: "少年天子，志在中兴" }
  - { id: general, name: 边关总兵, description: "手握重兵，进退两难" }
  - { id: eunuch, name: 司礼监秉笔, description: "权倾朝野，如履薄冰" }
  - { id: rebel, name: 义军首领, description: "揭竿而起，逐鹿天下" }
  - { id: secretary, name: 内阁首辅, description: "执笔天下，暗流涌动" }

# 事件调度
event_scheduler:
  max_active_events: 3
  random_check_per_time_unit: true
```

### chapters/ 定位

章节保留为"大幕"概念：
- 定义当前阶段的主题基调
- 限定可出场NPC范围
- 提供开场旁白
- 推进条件：min_turns/max_turns + Claude叙事成熟度判定（已实现）

章节不再承担事件触发职责，事件由时间系统和调度器独立驱动。

## 三、引擎运行流程

```
玩家发言
  │
  ├─→ 意图识别（DeepSeek）
  │     ├─ 有游戏行为 → Claude判定数值影响 → 实时结算
  │     └─ 纯闲聊 → 无结算
  │
  ├─→ 回合计数 +1
  │     └─ 达到 turns_per_unit？
  │           ├─ 是 → 时间推进
  │           │     ├─ 自然衰减
  │           │     ├─ 阈值检查 → 触发事件？
  │           │     ├─ 事件调度（timeline/random/disaster/npc/chain）
  │           │     └─ 活跃事件倒计时检查
  │           └─ 否 → 跳过
  │
  ├─→ 有新事件触发？
  │     └─ 是 → 以叙事方式插入对话（NPC闯入/急报/天象）
  │
  ├─→ 章节推进检查（min_turns ~ max_turns + Claude判定）
  │
  └─→ 生成回复（旁白/NPC对话）
        └─ 开头标注当前时间
```

## 四、从小说生成剧本的流程

1. **输入：** 小说文本（或大纲）
2. **AI提取（Claude）：**
   - 时间线：关键事件按时间排列
   - 人物：主要角色及其关系、动机
   - 冲突：核心矛盾和转折点
   - 数值维度建议：根据题材推荐4-6个维度
3. **输出：** 初版 manifest + timeline + events + npcs YAML
4. **人工校准：**
   - 调整数值平衡（初始值、衰减速率、阈值）
   - 确认哪些事件是锚点（不可改变）vs 可被玩家影响
   - 补充随机事件池
   - 设计分支结局
5. **测试迭代：** 试玩 → 调参 → 再试

## 五、与现有系统的关系

| 现有模块 | 改造方向 |
|----------|----------|
| progression.py | 保留章节推进逻辑，新增时间系统调用 |
| world_evolution.py | 改造为事件调度器 |
| game_master.py | 新增意图识别→数值结算流程 |
| state.py | 扩展：新增 game_time、active_events、event_history |
| narrator.py | 旁白模板加入时间标注 |
| client.py | 不变 |

**新增模块：**
- `src/engine/time_system.py` — 时间推进、衰减、倒计时
- `src/engine/event_scheduler.py` — 事件调度、条件匹配、优先级队列
- `src/engine/metric_engine.py` — 数值结算、阈值检查

## 六、MVP验证计划

用明末剧本验证核心循环：

1. 先实现时间系统 + 自然衰减 + 旁白带时间
2. 再实现事件调度器 + 3个timeline事件 + 5个随机事件
