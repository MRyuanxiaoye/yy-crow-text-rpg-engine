# 阶段1：排除引擎（Exclusion Engine）

## 在方法链中的位置

```
  阶段0：数据预处理与特征工程
        ↓
【阶段1：排除引擎】 ← 你在这里
        ↓
  阶段2：权重引擎
        ↓
  阶段3：组合生成器
        ↓
  阶段4：购买优化器
```

## 为什么是第二步（紧接在特征工程之后）？

### 起因
号码池很大（大乐透前区35个、双色球红球33个），如果直接对所有号码计算权重，噪声会很大——很多号码其实"明显不太可能出现"，但权重引擎无法有效区分它们，因为权重算法擅长的是**在合理候选中排序**，而不是**识别明显不合理的候选**。

举个例子：一个号码已经连续出现了3期，下一期继续出现的概率极低。但如果直接交给权重引擎，频率回归算法可能因为它"最近很热"而给它高权重，造成误判。

### 目的
在权重计算之前，先**缩小候选池**——把那些"大概率不会出现"的号码排除掉。这样做的好处：
1. **降噪**：权重引擎只需要在更小、更合理的候选池中工作，精度更高
2. **提效**：候选池缩小30%-50%，后续所有计算量相应减少
3. **互补**：排除引擎用的是"反向思维"（什么不会出现），与权重引擎的"正向思维"（什么会出现）形成互补

### 为什么不能放在权重引擎之后？
如果先算权重再排除，那排除就变成了"在权重结果上做过滤"，这会导致：
- 权重引擎在全量号码上计算，噪声大，权重本身就不准
- 排除变成了对不准的权重做二次加工，误差叠加
- 先排除再算权重，是"先粗筛再精排"的经典策略，信息损失最小

## 核心设计：多算法加权投票

排除引擎不是一个算法说了算，而是**6个排除算法各自独立判断**，最后通过加权投票决定。

```
                    ┌── 算法1.1：连续重复排除器 ──┐
                    ├── 算法1.2：遗漏值异常排除器 ─┤
特征矩阵 ──────────→├── 算法1.3：极端组合约束排除器 ┼──→ 加权投票融合 ──→ 排除号码集合
                    ├── 算法1.4：马尔可夫转移排除器 ─┤
                    ├── 算法1.5：周期性排除器 ────┤
                    └── 算法1.6：聚类异常排除器 ──┘
```

**为什么要多算法投票而不是用一个算法？**
- 单一算法只能从一个维度判断，容易有盲区
- 多算法从不同角度分析，互相补充
- 投票机制天然具有容错性：即使某个算法判断失误，只要多数算法正确，最终结果仍然正确
- 通过回测调整各算法权重，让表现好的算法话语权更大

## 算法1.1：连续重复排除器

### 设计起因
观察历史数据可以发现：一个号码连续出现2期是常见的，但连续出现3期及以上的概率急剧下降。这是因为在有限号码池中，每个号码的单期出现概率本身就不高（大乐透前区约 5/35 ≈ 14.3%），连续出现的概率是指数级衰减的。

### 目的
识别那些"已经连续出现多期"的号码，给予较高的排除置信度。

### 逻辑

```
输入：
  consecutive_count[v] = 号码 v 从最近一期往回连续出现的期数
  base_prob[v] = 号码 v 的历史单期出现概率

处理：
  如果 consecutive_count[v] == 0:
    该号码没有连续出现，不排除
    confidence = 0

  如果 consecutive_count[v] == 1:
    刚出现了1期，连续出现2期是常见的，轻微排除
    confidence = 0.1

  如果 consecutive_count[v] == 2:
    已经连续2期，继续出现的概率明显降低
    confidence = 0.4

  如果 consecutive_count[v] >= 3:
    连续3期及以上，极少见
    confidence = 0.7 + 0.1 * (consecutive_count[v] - 3)
    上限为 0.95

  更精确的公式：
    confidence = 1 - base_prob[v] ^ consecutive_count[v]
    含义：连续出现 N 次的概率 = p^N，排除置信度 = 1 - p^N

输出：
  每个号码的排除置信度（0~1之间）
```

### 边界情况
- 某些"超级热号"可能确实会连续出现多期，所以置信度不设为1.0，留有余地
- 该算法的权重在融合时不宜过高（建议0.15），避免过度排除热号

## 算法1.2：遗漏值异常排除器

### 设计起因
一个号码刚刚出现过（遗漏值很小），在短期内再次出现的概率相对较低。这不是说它"不可能"出现，而是从概率分布上看，大多数号码的出现间隔都大于1-2期。如果一个号码的平均间隔是7期，那它在出现后的第1-2期内再次出现，属于"异常短间隔"。

### 目的
识别那些"刚出现不久、遗漏值远低于平均间隔"的号码，给予排除建议。

### 逻辑

```
输入：
  missing[v] = 号码 v 的当前遗漏值（已经多少期没出现）
  avg_gap[v] = 号码 v 的历史平均间隔期

处理：
  ratio = missing[v] / avg_gap[v]

  如果 ratio < 0.3:
    遗漏值远低于平均间隔（刚出现不久）
    confidence = 0.6 * (1 - ratio / 0.3)
    含义：遗漏值越小（越刚出现），排除置信度越高

  如果 0.3 <= ratio < 1.0:
    遗漏值在正常范围内，不排除
    confidence = 0

  如果 ratio >= 1.0:
    遗漏值已达到或超过平均间隔，不排除（反而应该在权重引擎中加分）
    confidence = 0

输出：
  每个号码的排除置信度（0~1之间）
```

### 与算法1.1的关系
- 算法1.1关注"连续出现"（遗漏值=0的特殊情况）
- 算法1.2关注"遗漏值偏小"（更广泛的范围）
- 两者有部分重叠（连续出现的号码遗漏值一定为0），但角度不同
- 在融合时，两者的置信度会叠加，对"刚出现且连续出现"的号码排除力度更大

## 算法1.3：极端组合约束排除器

### 设计起因
这个算法与前两个不同——它不是看单个号码，而是看**号码组合的整体特征**。历史数据表明，某些组合特征几乎不会出现：
- 全奇数或全偶数的概率 < 2%
- 全大号或全小号的概率 < 2%
- 和值极端偏高或偏低的概率 < 5%
- 连号超过3组的概率 < 1%

如果某个号码的加入会导致组合"走向极端"，那这个号码就应该被标记风险。

### 目的
从组合层面约束号码选择，排除那些"会导致组合变得极端"的号码。这是其他5个算法都做不到的——它们都是从单个号码的角度分析，而这个算法从全局组合的角度分析。

### 逻辑

```
输入：
  combo_stats = 阶段0输出的组合特征历史统计
    包括：奇偶比分布、大小比分布、和值分布、连号分布、AC值分布
  current_context = 当前已知的约束信息

处理：
  这个算法不是直接给单个号码打分，而是计算"约束违反风险分数"

  步骤1：从历史数据中统计各组合特征的分布
    odd_even_dist = {
      "5:0": 0.01, "4:1": 0.15, "3:2": 0.35,
      "2:3": 0.33, "1:4": 0.14, "0:5": 0.02
    }
    （以大乐透前区5个号码为例）

  步骤2：对每个候选号码，评估它对组合特征的影响
    对于号码 v：
      假设 v 被选入，计算它对奇偶比、大小比的贡献
      如果 v 是奇数且当前已有较多奇数被选为高权重 → 风险升高
      如果 v 会导致和值超出 [均值-2σ, 均值+2σ] → 风险升高

  步骤3：综合风险评分
    risk_score = 加权求和(各约束的违反程度)
    confidence = sigmoid(risk_score - threshold)

输出：
  每个号码的"约束违反风险分数"（0~1之间）
  注意：这个分数是条件性的，取决于其他号码的选择情况
```

### 特殊性
- 这是唯一一个**组合层面**的排除算法，其他5个都是**号码层面**的
- 它的排除置信度不是绝对的，而是依赖于"其他号码的选择"
- 在融合时，它的权重最高（0.25），因为组合约束是最硬的规律
- 实现上比其他算法复杂，需要考虑号码之间的组合效应

### 与阶段3的关系
- 阶段3的"软约束检查器"也会检查组合特征，但那是在组合已经生成之后
- 阶段1的这个算法是在号码选择之前就做预判，提前排除"会导致极端组合"的号码
- 两者形成"前置预防 + 后置验证"的双重保障

## 算法1.4：马尔可夫转移排除器

### 设计起因
马尔可夫链的核心假设是：下一个状态只依赖于当前状态（一阶）或最近几个状态（高阶）。在开奖数据中，这意味着"上一期出了什么号码"对"下一期出什么号码"有一定的统计影响。通过构建转移矩阵，可以发现某些号码之间的"接力"关系很弱——即上一期出了号码A之后，下一期出号码B的概率极低。

### 目的
利用号码之间的转移概率，排除那些从上一期号码出发"几乎不可能转移到"的号码。

### 逻辑

```
输入：
  transition_matrix = 阶段0计算的一阶马尔可夫转移矩阵
    transition_matrix[i][j] = P(下一期出现j | 上一期出现i)
  last_draw = 上一期的开奖号码列表（如 [5, 12, 18, 23, 35]）

处理：
  对于每个候选号码 v：

    步骤1：计算从上一期每个号码到 v 的转移概率
      probs = [transition_matrix[i][v] for i in last_draw]

    步骤2：取平均转移概率
      avg_transition_prob = mean(probs)

    步骤3：与阈值比较
      threshold = 所有号码平均转移概率的 0.3 倍（即远低于平均水平）

      如果 avg_transition_prob < threshold:
        confidence = (1 - avg_transition_prob / threshold) * 0.8
        含义：转移概率越低于阈值，排除置信度越高
      否则:
        confidence = 0

输出：
  每个号码的排除置信度（0~1之间）
```

### 局限性
- 一阶马尔可夫假设较强，实际数据可能需要二阶甚至更高阶
- 转移矩阵需要足够多的历史数据才能稳定（至少200期以上）
- 数据量不足时，转移矩阵会很稀疏，此时该算法的权重应降低

### 与权重引擎的关系
- 排除引擎的马尔可夫排除器关注**低转移概率**（排除不可能的）
- 权重引擎的马尔可夫转移权重关注**高转移概率**（加权可能的）
- 两者共享同一个转移矩阵，但从相反方向使用它

## 算法1.5：周期性排除器

### 设计起因
某些号码的出现可能存在周期性模式。比如号码7可能每隔6-8期出现一次，形成一个大致的周期。如果我们能识别出这个周期，就可以判断：当前期距离号码7上次出现的期数，是否处于"周期窗口"内。如果不在窗口内，该号码出现的概率较低。

### 目的
通过频域分析（FFT/自相关）识别号码的出现周期，排除当前不在周期窗口内的号码。

### 逻辑

```
输入：
  对于每个号码 v：
    appearance_series[v] = 一个0/1序列，长度为总期数
      1表示该期出现了v，0表示没有
    （来自阶段0的时序特征）

处理：
  步骤1：周期检测
    方法A：自相关分析
      计算 appearance_series[v] 的自相关函数 ACF
      找到 ACF 的第一个显著峰值对应的滞后期数 → 主周期 T
      如果没有显著峰值 → 该号码无明显周期，跳过

    方法B：FFT频谱分析
      对 appearance_series[v] 做快速傅里叶变换
      找到频谱中的主频率 f → 主周期 T = 1/f
      如果主频率的振幅不显著 → 无明显周期，跳过

  步骤2：周期窗口判断
    last_appearance = 号码 v 上次出现的期数
    current_gap = 当前期 - last_appearance（即当前遗漏值）
    expected_next = T 的整数倍中最接近 current_gap 的值

    deviation = |current_gap - expected_next| / T

    如果 deviation > 0.4:
      当前不在周期窗口内，排除置信度升高
      confidence = min(0.7, deviation * 0.8)
    否则:
      在周期窗口内或接近窗口，不排除
      confidence = 0

输出：
  每个号码的排除置信度（0~1之间）
  注意：无明显周期的号码，confidence 固定为 0（不参与排除）
```

### 局限性
- 并非所有号码都有明显周期，很多号码的出现是近似随机的
- 周期可能不稳定，会随时间漂移
- 因此该算法的融合权重最低（0.10），仅作为辅助参考
- 需要足够长的历史数据（至少100期）才能做有意义的周期检测

### 独特价值
- 这是唯一一个从**频域**角度分析的算法，其他算法都在时域工作
- 即使权重低，它提供了一个完全不同的分析维度，有助于提高整体的多样性和鲁棒性

## 算法1.6：聚类异常排除器

### 设计起因
将每一期的开奖号码视为一个多维向量（如大乐透前区是5维向量），所有历史开奖就构成了一个向量集合。对这个集合做聚类分析，可以发现历史开奖号码倾向于聚集在某些"区域"。如果一个候选号码会导致组合落在所有聚类中心都很远的位置，说明这种组合在历史上极少出现，属于"异常"。

### 目的
从高维空间的角度，排除那些会导致组合"偏离历史分布"的号码。

### 逻辑

```
输入：
  historical_vectors = 所有历史开奖号码组合的向量表示
    如大乐透前区：[[5,12,18,23,35], [2,7,15,28,31], ...]
  cluster_model = 对 historical_vectors 做 K-means 聚类的结果
    包括 K 个聚类中心

处理：
  步骤1：聚类（在阶段0中预计算，这里直接使用）
    使用 K-means（K=5~10，通过肘部法则确定）
    得到 K 个聚类中心 centers = [c₁, c₂, ..., cₖ]

  步骤2：对每个候选号码评估异常度
    对于号码 v：
      假设 v 被选入当前组合（与其他高权重号码组合）
      构造一个"假设组合向量" hypothetical_combo

      计算 hypothetical_combo 到最近聚类中心的距离：
        min_distance = min(euclidean(hypothetical_combo, cᵢ) for cᵢ in centers)

      计算异常分数：
        avg_distance = 所有历史组合到其最近聚类中心的平均距离
        anomaly_score = min_distance / avg_distance

      如果 anomaly_score > 2.0:
        距离是平均值的2倍以上，明显异常
        confidence = min(0.8, (anomaly_score - 2.0) * 0.3 + 0.3)
      否则:
        在正常范围内
        confidence = 0

输出：
  每个号码的排除置信度（0~1之间）
```

### 与算法1.3的区别
- 算法1.3（极端组合约束排除器）检查的是**具体的组合特征**（奇偶比、和值等）
- 算法1.6（聚类异常排除器）检查的是**高维空间中的整体分布**
- 算法1.3是"规则驱动"的（人工定义什么是极端）
- 算法1.6是"数据驱动"的（让聚类算法自动发现什么是正常）
- 两者互补：算法1.3捕捉已知的极端模式，算法1.6捕捉未知的异常模式

## 融合器：加权投票

### 设计起因
6个算法各自从不同角度给出排除建议，但它们的可靠性不同。有些算法（如极端组合约束）基于非常硬的统计规律，可靠性高；有些算法（如周期性排除）基于较弱的假设，可靠性低。需要一个融合机制来综合所有算法的意见。

### 目的
将6个算法的排除置信度融合为一个最终的排除决策。

### 逻辑

```
输入：
  对于每个号码 v，6个算法各自输出一个排除置信度：
    confidences[v] = {
      "consecutive":   c₁,
      "missing_value": c₂,
      "extreme_combo": c₃,
      "markov":        c₄,
      "periodicity":   c₅,
      "cluster":       c₆,
    }

  算法权重（通过历史回测确定）：
    weights = {
      "consecutive":   0.15,
      "missing_value": 0.20,
      "extreme_combo": 0.25,  ← 最高，因为组合约束最硬
      "markov":        0.15,
      "periodicity":   0.10,  ← 最低，因为周期性假设最弱
      "cluster":       0.15,
    }

处理：
  步骤1：加权求和
    final_confidence[v] = Σ(weights[algo] * confidences[v][algo])

  步骤2：应用排除阈值
    EXCLUSION_THRESHOLD = 0.6（可调参数）

    如果 final_confidence[v] > EXCLUSION_THRESHOLD:
      号码 v 被排除
    否则:
      号码 v 保留，进入权重引擎

  步骤3：排除比例检查
    MAX_EXCLUSION_RATE = 0.50（硬性上限）

    如果排除比例超过上限：
      按 final_confidence 降序排列
      只排除前 MAX_EXCLUSION_RATE * total_numbers 个号码
      含义：即使很多号码的置信度都超过阈值，也不能排除太多

输出：
  excluded_numbers = 被排除的号码列表
  remaining_numbers = 保留的号码列表（传入阶段2）
  exclusion_details = 每个被排除号码的详细原因
```

### 权重调优
- 初始权重是人工设定的经验值
- 通过历史回测自动调优：
  - 目标：最小化 `exclusion_miss_rate`（实际出现的号码被错误排除的比例）
  - 方法：Optuna 贝叶斯优化，搜索6个权重参数和排除阈值
  - 约束：所有权重之和为1，每个权重在 [0.05, 0.40] 范围内

### 核心指标
- **排除精度**：被排除的号码中，确实没出现的比例（越高越好，目标 > 85%）
- **误排率**：实际出现的号码被错误排除的比例（越低越好，目标 < 15%）
- **排除比例**：被排除号码占总号码的比例（目标 30%-50%）
